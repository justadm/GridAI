# Инцидент 2026-03-30: `gridai.ru` отдавал `502 Bad Gateway`

## Кратко

- Дата выявления: `2026-03-30`
- Внешний симптом: `https://gridai.ru/` отвечал `HTTP/2 502`
- Точка отказа: хостовый `nginx` на прод-сервере не мог достучаться до upstream `127.0.0.1:13001`
- Фактическая причина: контейнеры `gridai-web` и `gridai-bot` уходили в бесконечный restart loop

## Что произошло

Продовый `nginx` проксирует основной домен `gridai.ru` на локальный SSR/API процесс по адресу `127.0.0.1:13001`.
Во время инцидента на этом порту не было слушателя, поэтому `nginx` возвращал `502`.

Проверка на сервере показала:

- `curl -sI http://127.0.0.1:13001/` завершался с ошибкой подключения
- `ss -ltnp` не показывал listener на `127.0.0.1:13001`
- `docker ps` показывал `gridai-web` и `gridai-bot` в статусе `Restarting (1)`
- логи обоих контейнеров содержали одну и ту же ошибку: `Bot already running with pid 18. Exiting.`

## Корневая причина

Проблема была в bootstrap-логике процесса в [`src/index.js`](../src/index.js).

До исправления:

- `ensureSingleInstance()` вызывался безусловно еще до чтения `DISABLE_BOT`
- контейнер `web` стартует через тот же `src/index.js`, но с `DISABLE_BOT=true`
- несмотря на это, `web` тоже создавал и проверял bot PID lock в `/tmp/gridai-bot.pid`
- после рестарта контейнера PID внутри контейнера мог снова быть `18`
- код видел файл `/tmp/gridai-bot.pid` со значением `18`, делал `process.kill(18, 0)` и успешно находил текущий процесс
- текущий процесс ошибочно считался "уже запущенным ботом" и завершался с кодом `1`

Итог:

- `gridai-web` не доходил до `app.listen(...)`
- порт `127.0.0.1:13001` не поднимался
- внешний `nginx` возвращал `502`

## Почему баг появился

- использовалась общая entrypoint-логика для `bot` и `web`
- блокировка одиночного процесса была привязана к файлу PID, но не к факту запуска именно bot-режима
- проверка PID не различала "другой живой процесс" и "это текущий процесс после рестарта контейнера"
- не было отдельной smoke-проверки, которая бы подтверждала, что `web` реально слушает `127.0.0.1:13001` после релиза

## Исправление

Исправление внесено в [`src/index.js`](../src/index.js):

- `ensureSingleInstance()` больше не вызывается глобально при старте любого процесса
- PID lock теперь создается только внутри ветки, где `DISABLE_BOT != true`
- добавлена защита от ложного срабатывания: PID, равный `process.pid`, не считается конфликтом

После фикса:

- `gridai-web` успешно поднялся и начал слушать `http://127.0.0.1:13001`
- `https://gridai.ru/` снова начал отвечать `200 OK`

## Как выявлять в дальнейшем

Минимальный чек при жалобе на `502`:

```bash
curl -skI https://gridai.ru/
ssh msk curl -sI http://127.0.0.1:13001/
ssh msk docker ps --format '{{.Names}}\t{{.Status}}\t{{.Ports}}' | rg '^gridai-'
ssh msk docker logs --tail 100 gridai-web
ssh msk docker logs --tail 100 gridai-bot
ssh msk ss -ltnp | rg '127.0.0.1:13001'
```

Сигналы, что это именно этот класс отказа:

- снаружи `502`, а `/healthz` у внешнего `nginx` при этом отвечает `200`
- на сервере нет listener на `127.0.0.1:13001`
- `gridai-web` в `Restarting`
- в логах есть `Bot already running with pid`

## Меры предотвращения

### 1. Код

- не выполнять bot-specific lock до вычисления `DISABLE_BOT`
- не использовать общий PID lock для разных режимов процесса без явной изоляции
- при использовании PID-файлов всегда учитывать кейс self-PID и stale PID

### 2. Наблюдаемость

- добавить внешний алерт на `https://gridai.ru/` с ожиданием не только TCP/TLS, но и `HTTP 200/301`
- добавить отдельную проверку upstream: `http://127.0.0.1:13001/`
- мониторить restart count контейнеров `gridai-web` и `gridai-bot`
- алертить на отсутствие listener на `127.0.0.1:13001`

### 3. Релизный процесс

- после каждого деплоя проверять:
  - `docker ps` для `gridai-web` и `gridai-bot`
  - `curl -sI http://127.0.0.1:13001/`
  - `curl -skI https://gridai.ru/`
- не считать деплой успешным, пока `web` не начал слушать целевой порт

### 4. Тесты

- добавить регрессионный тест на bootstrap:
  - при `DISABLE_BOT=true` bot PID lock не должен создаваться и проверяться
  - при наличии PID, равного текущему `process.pid`, процесс не должен завершаться с ошибкой

## Что еще стоит сделать

- вынести web и bot в разные entrypoint-файлы, чтобы убрать скрытую связанность bootstrap-логики
- оформить отдельный ops-runbook по прод-проверкам после деплоя
- привести документацию к одному способу прод-запуска: сейчас в репозитории есть и старый `systemd` сценарий, и актуальный `docker compose` сценарий
